# MySQL 子查询优化（5.7 / 8.0 通用）

---

## MYSQL-SUBQUERY-001：IN 子查询 → EXISTS 改写

**严重性**：⚠️ 警告

**条件**：外表行数 < 10000 且内表行数 > 1000 时。

```sql
-- ❌ 原写法
SELECT * FROM orders WHERE user_id IN (SELECT id FROM users WHERE status = 'active');

-- ✅ 改写
SELECT * FROM orders o WHERE EXISTS (
    SELECT 1 FROM users u WHERE u.id = o.user_id AND u.status = 'active'
);
```

**风险**：子查询结果集很小（<100 行）时 EXISTS 可能比 IN 慢。

**证据**：MySQL 8.0 Reference Manual §8.2.2.1 Semijoin Transformations

---

## MYSQL-SUBQUERY-002：EXISTS → INNER JOIN 改写

**严重性**：⚠️ 警告

**条件**：小结果集 + 需要子查询字段时。

```sql
-- ❌
SELECT * FROM orders o WHERE EXISTS (
    SELECT 1 FROM users u WHERE u.id = o.user_id AND u.level = 'VIP'
);

-- ✅
SELECT o.* FROM orders o
INNER JOIN users u ON u.id = o.user_id AND u.level = 'VIP';
```

**风险**：可能产生重复行，需加 DISTINCT。

## MYSQL-SUBQUERY-003：NOT IN → NOT EXISTS 改写

**严重性**：⚠️ 警告

**核心陷阱**：子查询返回 NULL 时 `NOT IN` 永远返回空。

```sql
-- ❌ 如果 users.status 有 NULL，结果永远是空
SELECT * FROM orders WHERE user_id NOT IN (SELECT id FROM users);

-- ✅ 防 NULL 陷阱
SELECT * FROM orders o WHERE NOT EXISTS (
    SELECT 1 FROM users u WHERE u.id = o.user_id
);
```

**证据**：MySQL 8.0 Reference Manual §8.2.2.3

---

## MYSQL-SUBQUERY-004：子查询 → 派生表 + JOIN

**严重性**：ℹ️ 提示

**条件**：MySQL 5.7 优化器不物化子查询，FROM 子查询（派生表）物化能力弱，大结果集易临时表落盘。

```sql
-- ❌ MySQL 5.7 大结果集时性能差
SELECT * FROM (
    SELECT user_id, COUNT(*) AS cnt FROM orders GROUP BY user_id
) AS t WHERE cnt > 10;

-- ✅ 改写为 JOIN
SELECT o.* FROM orders o
INNER JOIN (
    SELECT user_id FROM orders GROUP BY user_id HAVING COUNT(*) > 10
) t ON o.user_id = t.user_id;
```

## MYSQL-SUBQUERY-005：SELECT 子查询 → LEFT JOIN

**严重性**：🔴 严重

SELECT 列表中每行执行一次子查询，N+1 问题。

```sql
-- ❌ N+1 查询
SELECT u.*, (SELECT COUNT(*) FROM orders WHERE user_id = u.id) AS order_count
FROM users u;

-- ✅ LEFT JOIN + GROUP BY
SELECT u.*, COUNT(o.id) AS order_count
FROM users u
LEFT JOIN orders o ON o.user_id = u.id
GROUP BY u.id;
```

**风险**：可能改变 NULL 行为（LEFT JOIN 可能返回 NULL）。

**证据**：MySQL 8.0 Reference Manual §8.2.2.4
