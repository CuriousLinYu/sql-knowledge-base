# MySQL 索引优化（5.7 / 8.0 通用）

---

## MYSQL-INDEX-001：联合索引最左前缀原则

**严重性**：⚠️ 警告

WHERE/ORDER BY 必须从联合索引的第一列开始**连续**使用，跳过列会导致后续列不走索引。

```sql
-- 索引: (a, b, c)
-- ✅ 走索引: a / a,b / a,b,c
-- ❌ 不走索引: b / c / a,c（跳过了 b）
```

**证据**：MySQL 8.0 Reference Manual §8.3.6 Multiple-Column Indexes

## MYSQL-INDEX-002：索引列上使用函数 → 索引失效

**严重性**：🔴 严重

```sql
-- ❌ 函数包裹索引列
SELECT * FROM orders WHERE DATE(created_at) = '2024-01-01';
SELECT * FROM users WHERE LOWER(username) = 'admin';

-- ✅ 改为范围查询
SELECT * FROM orders WHERE created_at >= '2024-01-01' AND created_at < '2024-01-02';
```

**证据**：MySQL 8.0 Reference Manual §8.3.1 How MySQL Uses Indexes

## MYSQL-INDEX-003：前缀模糊查询 → 索引失效

**严重性**：⚠️ 警告

```sql
-- ❌ 前缀模糊，索引失效
SELECT * FROM products WHERE name LIKE '%phone';
SELECT * FROM products WHERE name LIKE '%phone%';

-- ✅ 前缀匹配，走索引
SELECT * FROM products WHERE name LIKE 'phone%';
```

**方案**：后缀/中间匹配用全文索引 + 生成列反转。见 `深度方案/全文索引与生成列.md`。

**证据**：MySQL 8.0 Reference Manual §8.3.1

## MYSQL-INDEX-004：覆盖索引——避免回表

**严重性**：ℹ️ 提示

所有查询列都在索引中，EXPLAIN Extra 显示 `Using index`，无需回表。

```sql
-- 索引: (user_id, status, created_at)
-- ✅ 覆盖索引，Extra: Using index
SELECT user_id, status, created_at FROM orders WHERE user_id = 1;

-- ❌ 需要回表
SELECT * FROM orders WHERE user_id = 1;
```

**详见**：`深度方案/覆盖索引.md`

## MYSQL-INDEX-005：OR 条件不全部走索引 → 全表扫描

**严重性**：⚠️ 警告

OR 连接的条件必须**全部**都有索引可用，否则全表扫描。

```sql
-- ❌ status 有索引，amount 无 → 全表扫描
SELECT * FROM orders WHERE status = 'pending' OR amount > 1000;

-- ✅ 改为 UNION ALL
SELECT * FROM orders WHERE status = 'pending'
UNION ALL
SELECT * FROM orders WHERE amount > 1000;
```

**证据**：MySQL 8.0 Reference Manual §8.3.1
